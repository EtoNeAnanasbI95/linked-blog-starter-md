# Коллекции
## 📌 Основные интерфейсы коллекций в .NET
1. **`IEnumerable<T>`**
    - Самый базовый.
    - Декларирует только один метод: `GetEnumerator()` → позволяет _перебирать_ элементы (`foreach`).
    - Никакой информации о размере, доступе по индексу, добавлении/удалении.
    - Это «последовательность для чтения».
2. **`ICollection<T>`**
    - Наследует `IEnumerable<T>`.
    - Добавляет:
        - `int Count { get; }` — количество элементов;
        - `bool IsReadOnly { get; }`;
        - Методы для изменения коллекции:
			```
			void Add(T item);
			void Clear();
			bool Contains(T item);
			void CopyTo(T[] array, int arrayIndex);
			bool Remove(T item);
			
			```
	    - То есть `ICollection<T>` = «последовательность с размером и базовыми методами для работы с элементами».
	    - Всё ещё **не гарантирует индексацию** (это делает уже `IList<T>`).
3. **`IList<T>`**
    - Наследует `ICollection<T>`.
    - Добавляет доступ по индексу (`this[int index]`), `Insert`, `RemoveAt`.
    - Это «упорядоченная коллекция с индексами».
    - `List<T>` реализует `IList<T>`.
## 📌 Как это выглядит в иерархии
```
IEnumerable<T>
   └── ICollection<T>
         └── IList<T>
               └── List<T>
```
## 📌 Где применять
- **`IEnumerable<T>`** → когда нужен только перебор (_read-only, самая общая абстракция_).
- **`ICollection<T>`** → когда тебе важно знать размер и возможность добавлять/удалять, но индексация не нужна.
- **`IList<T>`** → когда нужна индексация и порядок (например, работа с UI-таблицей).
- **`List<T>`** → конкретная реализация.
# LINQ
## Разница между select и where
- Select нужен для изменения данных в коллекции, а where для их фильтрации. То есть если в where в лямбду мы пропишем просто условие, которое должно выполняться для итерируемого объекта, то в селект мы напишем что надо сделать с этим объектом
```csharp
List<int> numbers = new List<int> { 1, 2, 3, 4, 5 };
var bigNumbers = numbers.Where(x => x > 3); // Вернёт 4, 5

List<int> numbers = new List<int> { 1, 2, 3, 4, 5 };
var doubledNumbers = numbers.Select(x => x * 2); // Вернёт 2, 4, 6, 8, 10
```
# Expression
В сути своей любой делегат это экспрешон. Типа когда мы пишем `x => x.Filed != constant` мы пишем в сути экспрешон, который имеет подтип Func<T, bool>, но им не является. В чём фишка? 
Когда мы пишем Func<T, bool> и делаем из этого переменную компилятор делает переменную **делегатом**, то есть объектом который указывает на какую-то готовую скомпилированную функцию. То есть в Func нельзя заглянуть, только запустить его, он как некий чёрный ящик которым можно воспользоваться но заглянуть в который нельзя.
Expression отличается от Func как раз тем, что **он НЕ КОМПИЛИРИУЕТСЯ** в функцию, он хранит в себе **описание логических действий, с которыми можно что-либо делать**. 
`Expression<Func<T, bool>>` — это **не исполняемый код**, а **структура данных, описывающая этот код** в виде **дерева выражений**.
Например:
`Expression<Func<int, bool>> expr = x => x % 2 == 0;`
Компилятор не создаёт исполняемый метод, а **строит объектное дерево**:
```shell
Lambda
 └── BinaryExpression (%)
       ├── Parameter (x)
       ├── Constant (2)

```
Это как раз та причина, почему работает EF. Потому что иногда нужно **не выполнять** код, а **анализировать его структуру**.
Например:
`db.Users.Where(u => u.Age > 18)`
Если бы `Where` принимал `Func<User, bool>`, то:
- CLR бы **сразу выполнил** лямбду;
- EF **не узнал бы, что ты имел в виду `u.Age > 18`**.
А с `Expression<Func<User, bool>>`:
- EF получает **описание выражения** (дерево),
- Может разобрать `u.Age > 18`,
- И перевести в SQL:
```sql
SELECT * FROM Users WHERE Age > 18
```
