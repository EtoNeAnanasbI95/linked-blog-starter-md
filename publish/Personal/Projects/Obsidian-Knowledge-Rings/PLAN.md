## 1. Инициализируй основу плагина Obsidian с React UI
Сделай базовый плагин для Obsidian на TypeScript с поддержкой React в custom view.
Что нужно:
- TypeScript
- Obsidian plugin API
- React
- ReactDOM
- esbuild или стандартный шаблон сборки для Obsidian plugin
- отдельная custom view через `ItemView`
- settings tab
- commands registration
Ожидаемый результат:
- плагин собирается
- в Obsidian появляется новая view
- внутри view рендерится React-приложение
## 2. Подключи UI и графические библиотеки для будущих представлений
Добавь библиотеки, которые понадобятся для кольца и веток.
Список библиотек:
- `react`
- `react-dom`
- `reactflow`
- `clsx`
- `zod`
- `nanoid`
- `framer-motion`
- `tailwind`
Не добавляй тяжёлые layout-библиотеки на этом этапе, если сначала будешь делать layout вручную.
Ожидаемый результат:
- зависимости установлены
- React Flow успешно импортируется
- проект собирается без ошибок
## 3. Спроектируй доменную модель данных плагина
Создай типы данных для категорий, веток, звеньев и ссылок на md-файлы.
Нужно описать типы:
- `KnowledgeCategory`
- `KnowledgeBranch`
- `KnowledgeNode`
- `KnowledgeNodeStatus`
- `KnowledgeNodeContentRef`
- `KnowledgeGraphSnapshot`
Пример смысла:
- категория = сектор кольца
- ветка = набор звеньев внутри категории
- звено = задача или ссылка на заметку
- заметка = md-файл с произвольным содержимым
Ожидаемый результат:
- есть отдельный файл `types.ts`
- все базовые сущности типизированы
- типы готовы к парсингу и UI
## 4. Определи формат хранения данных в markdown-файлах
Придумай и реализуй минимальный и понятный формат, по которому плагин будет читать категории и звенья из заметок.
Нужно выбрать один формат и зафиксировать его:
- либо YAML frontmatter
- либо markdown-списки с inline metadata
- либо гибрид
Рекомендуемая структура:
- отдельная заметка категории
- внутри категории список узлов
- узел может ссылаться на другой `.md`
- узел может иметь `parent`, `branch`, `status`, `order`, `notePath`
Ожидаемый результат:
- есть документированный формат данных
- можно создать 2–3 тестовые markdown-заметки и распарсить их предсказуемо
## 5. Реализуй parser, который читает markdown-файлы Obsidian и строит внутреннюю модель
Напиши сервис, который сканирует выбранную папку или набор файлов и строит объект графа категорий и веток.
Что должен делать parser:
- читать markdown-файлы из vault
- распознавать категории
- распознавать узлы внутри категории
- собирать связи parent-child
- прикреплять путь до заметки, если узел ссылается на отдельный `.md`
- валидировать данные через `zod`
Ожидаемый результат:
- есть класс вроде `KnowledgeRepository` или `KnowledgeParser`
- на выходе получаешь нормализованный snapshot данных
- ошибки в данных не ломают плагин, а логируются
## 6. Сделай слой нормализации и индексации данных

Ожидаемый результат:

Ожидаемый результат:
